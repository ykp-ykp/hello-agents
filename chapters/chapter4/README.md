## 本章节主要实现了三种经典的智能体构建范式

ReAct、Plan-and-Solve、Reflection

### ReAct

**核心思想**：Thought-->Action-->Observation-->Thought-->.....

**思想解释**：使用系统提示此约束LLM每次思考都会吐出一个Thought和Action，Agent会去执行具体的Action，并将执行返回的结果作为Observation在下一轮循环喂给LLM。就这样依次循环，直到LLM认为当前可以给出答案为止。这个循环也叫做Agent Loop

**优点**：解释性强，可以看到LLM每次的思考过程。

**缺点**：没有强约束，完全依赖历史执行信息和LLM的自由发挥，容易产生幻觉。

### Plan-and-Solve

**核心思想**：是先让LLM生成一份任务计划书，后面再让Agent按照任务计划书一步一步的完成任务。

**需要构建两套提示词**：

- 第一套提示词是Plan：该提示词负责拆分任务，让LLM根据用户的问题拆分成具体的小步骤，如【step1，step2，step3】
- 第二套提示词是Solve：该提示词需要包含原始问题、完整的计划任务书、历史执行的步骤和结果、当前需要执行的步骤。在这个提示词的驱动下，LLM会参考历史执行步骤，并主要解决当前需要执行的步骤。

**需要构建一个历史记录history**：用于记录每一步调用LLM的**执行步骤**和**执行结果**，用于在后面的步骤中喂给LLM。

优点：步骤清晰，LLM会遵循事先的Plan计划去执行，可以有效避免中间步骤产生的幻觉；是和复杂任务。

缺点：对于没必要做Plan计划的简单任务，会增加token的消耗；强依赖计划的输出。

思考：该范式的构建重点是提示词的构建，需要将提示词拆分成两套，一套是Plan，一套是Solve。在此之上需要实现若干函数去辅助填充提示词中的占位内容。

### Reflection

**核心思想**：让LLM懂得自我反思，回答问题-->反思-->优化-->反思-->优化.....

**思想解释**：当LLM根据问题给出回答之后，该回答并不作为最终答案；而是先让LLM根据问题和它的回答内容进行反思，输出一段反思/审查；之后让LLM根据原始问题、反思/审查去进一步优化答案；agent会不断驱动LLM去执行反思-->优化这个流程，直到LLM认为回答已是最优。

**需要构建三套提示词**：

- 第一套提示词是初始提示词：很常规的提示此，让LLM根据问题直接回答即可
- 第二套提示词是反思提示词：让LLM作为一个审查专家，把原始问题、回答结果都给它，让它找问题，判断哪里可以优化。
- 第三套提示词是优化提示词：把原始问题、上次的回答结果、优化意见都给LLM，让LLM基于上述优化意见进一步给出答案。

优点：这种方式可以提升LLM输出答案的质量，让结果更优。

缺点：回答任何问题都会经过反思和优化两个过程，token消耗会翻倍；反思和优化二者是串行的，效率比普通不反思的agent更低。

### 思考

看起来Agent的不同范式主要区别是提示词的设计不同，以及为了满足不同提示词的要求而实现的一套"辅助代码"机制，这两者结合起来就变成了Agent。而Agent的主要作用是通过提示词去约束、指导LLM的思考和解决问题的过程。
